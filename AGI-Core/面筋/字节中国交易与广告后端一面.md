1.go的slice的扩容策略（大于256 n += (n+3*256)/4）  
答：slice 本质上是对底层数组的一个描述结构，包含指针、长度和容量。扩容时会申请新数组并拷贝旧数据；Go 新版本里小容量通常按 2 倍增长，大于一定阈值后增长比例会下降，目的是平衡扩容次数和内存浪费。

**2.go的new和make的区别（new返回指针 只分配零值内存并不初始化，make返回类型本身 分配内存并初始化）**  
答：`new(T)` 返回 `*T`，只做零值内存分配；`make` 只用于 `slice/map/channel`，返回的是初始化后的类型本身。一个偏通用分配，一个偏这三类引用类型的可用初始化。

3.向一个关闭的chan中读取会怎么样  
答：可以继续读。若缓冲区里还有数据，会先把剩余数据读完；读空之后会立刻返回该类型零值，并且 `ok=false`。

4.向已经关闭的chan中写入数据会怎么样？（panic）  
答：会直接 panic，所以 channel 的关闭一定要明确所有权。通常是谁创建、谁负责关闭，消费者一般不主动关。

5讲解一下输入url到网址用到了什么相关的协议介绍一下  
追问 http建立连接的时候会不会绑定端口！！！（随机绑定socket 这个有点不太熟悉）  
答：整体流程一般是浏览器检查缓存、DNS 解析域名、建立 TCP 连接、如果是 HTTPS 再做 TLS 握手，然后发 HTTP 请求，服务端处理后返回响应。  
客户端发起 TCP 连接时会为本地 socket 分配一个临时端口，所以是会绑定本地端口的，只是不是业务固定端口，而是系统动态分配的 ephemeral port。

6.介绍一下TCP三次握手和四次挥手  
追问 为什么TCP是3次连接 4次断开 （防止历史连接....，同步双方的序列号）  
答：三次握手是为了双方确认彼此的收发能力和初始序列号，避免历史报文造成错误连接；四次挥手则是因为 TCP 全双工，双方的发送和接收关闭要分别确认，所以通常需要四次。  
如果第二次和第三次确认能合并，某些场景看起来像“三次挥手”，但标准描述还是四次。

7.进程间的通信有哪些方式？什么场景使用  
答：常见有管道、消息队列、共享内存、信号量、信号、socket、mmap。  
本机父子进程简单通信常用管道，跨进程高性能共享数据常用共享内存，跨机器或跨语言一般更常用 socket/RPC。

8.线程和协程的区别  
答：线程是内核调度单位，协程是用户态轻量执行单元。协程切换成本更低，适合大量 IO 并发；线程隔离更天然，但调度和切换成本更高。

9.线程间的通信方式（有点懵逼，不过说了共享内存）  
答：线程间最常见的是共享内存 + 同步原语，也就是锁、条件变量、信号量。  
在 Go 里还会用 channel 做通信，把共享内存问题尽量转化成消息传递问题。

10.Redis 的持久化机制（说了一半最后自己把自己套进去了 不过不影响）  
答：Redis 主要是 RDB 和 AOF。RDB 适合做快照恢复，AOF 适合减少数据丢失窗口；线上常常一起开。

Redis如何实现分布式锁？  
答：核心是 `SET key value NX PX ttl` 加锁，value 要唯一，解锁要用 Lua 脚本做“比较后删除”。如果业务执行时间可能超过 ttl，还要做续约，也就是看门狗机制。

11.介绍一下MySQL的隔离级别  
答：读未提交、读已提交、可重复读、串行化。InnoDB 默认可重复读，兼顾一致性和并发性。

12.讲一下MySQL的悲观锁和乐观锁  
答：悲观锁是假设冲突会发生，所以先锁再操作，比如 `select ... for update`；乐观锁是假设冲突少，更新时再校验版本号或条件。互联网业务里乐观锁常用于高并发更新，悲观锁常用于强一致扣减。

13.一条更新的sql语句是怎么执行的  
答：先经过连接器、解析器、优化器、执行器，进入存储引擎后定位记录并加锁，修改 Buffer Pool 中的数据页，同时写 undo log 和 redo log；提交时再写 binlog 并完成两阶段提交。
