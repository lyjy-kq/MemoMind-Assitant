# golang：
1：切片和数组的区别  
答：数组长度是类型的一部分，大小固定，值语义更强；切片是对数组的一层描述，包含指针、长度和容量，支持动态扩容，更适合业务开发。传 slice 更轻，传 array 往往会复制整块数据。

2：切片底层  
答：slice 底层是一个三元组：指向底层数组的指针、len、cap。切片扩容时可能会申请新数组并拷贝旧数据，所以要注意切片共享底层数组带来的副作用。

3：讲一下channel  
答：channel 是 goroutine 之间通信和同步的原语，支持无缓冲和有缓冲两种模式。无缓冲更强调同步交接，有缓冲适合削峰和异步处理。

4: csp模型跟传统的共享内存有什么不一样  
答：CSP 强调通过通信共享内存，把并发协调更多落到消息传递上；传统共享内存则更依赖锁来保护共享状态。前者更容易控制数据流，后者在复杂场景下更容易出现锁竞争和死锁。

5：讲一下gmp模型  
答：G 是 goroutine，M 是内核线程，P 是执行 goroutine 所需的处理器上下文。调度器通过 P 把大量 G 高效映射到少量 M 上执行，从而在保持高并发的同时降低线程成本。

6：m跟p是1对1吗  
多个m竞争一个空闲p
一个p同一时间只绑定一个m  
答：同一时刻一个 P 只能绑定一个 M，但整体上不是全局永久 1 对 1。多个 M 可以竞争空闲 P，P 的数量通常接近可并行执行的逻辑核数。

# Mysql:
1.讲一下mvcc的原理  
答：MVCC 通过版本链、undo log 和 Read View 让读操作读到合适的历史版本，从而减少读写冲突。它的重点不是“没有锁”，而是让一致性读不必总跟写操作硬碰硬。

2.mysql的事务四大特性，怎么实现的  
答：ACID 分别是原子性、一致性、隔离性、持久性。原子性靠 undo log，持久性靠 redo log，一致性靠事务机制整体保证，隔离性靠锁和 MVCC。

3.假如高并发，你怎么从锁机制去优化  
优先用行锁，避免表锁，减少锁的持有时间
优化索引，无效索引会变成表锁
用rc  
答：这个方向是对的，但可以答得更完整：  
1. 尽量命中索引，让锁范围从“大扫一片”变成“精确锁行”；  
2. 缩短事务时间，把网络调用和复杂计算移出事务；  
3. 控制访问顺序，降低死锁概率；  
4. 根据业务容忍度选隔离级别，比如某些场景 RC 比 RR 并发更友好。  

4.行锁和表锁讲一下  
答：行锁粒度细，并发高，但管理开销更大；表锁粒度粗，并发低，但实现简单。InnoDB 主要用行锁，MyISAM 更偏表锁。

5.innodb和myisam的使用场景  
答：绝大多数在线业务优先选 InnoDB，因为它支持事务、行锁和崩溃恢复。MyISAM 现在更多是历史知识点，读多写少且不要求事务的旧场景才可能提到。

# kafka
1.说一下kafka的主题  
答：topic 是逻辑消息分类，一个 topic 会分成多个 partition。分区是 Kafka 并行和扩展的基本单位。

2.kafka怎么从保证消息有序性的（从消费者，话题队列本身，生产者去说）  
答：Kafka 只能保证分区内有序。要保证业务有序，一般要让同一个业务 key 始终发往同一分区，生产端按 key 路由，消费端顺序消费该分区，避免同一 key 被并行处理打乱。

3.你怎么应对消息堆积  
答：先看是生产速率过高、消费逻辑太慢，还是下游故障导致阻塞。常见手段有扩消费者、增分区、批量消费、异步化慢操作、消息降级和旁路补偿。

4.消息可靠  
答：要从生产、存储、消费三段看。生产端要有重试和 ack 策略，Broker 侧要有副本和持久化，消费端要做好幂等和 offset 提交时机控制，避免“处理失败但 offset 已提交”。

5.发散题：如何设计一个高可用的消息队列  
答：可以按写入、存储、消费、运维四层讲。写入层做负载均衡和限流，存储层做分区副本和故障转移，消费层做消费者组和重试死信，运维层做监控、扩缩容和数据保留策略。

# Redis
1.缓存雪崩除了时间过期随机化，还有什么办法？

我说限流，他说限流无脑，除了这个还有吗?

它提示从业务代码上解决  
答：可以从“让缓存不要同时失效”和“失效后不要同时打数据库”两头处理。除了过期时间打散，还可以用多级缓存、热点 key 永不过期 + 异步刷新、singleflight/互斥重建、预热缓存、兜底默认值和降级逻辑。  
如果面试官强调“从业务代码上解决”，那就重点讲互斥重建和异步刷新，而不是只说网关限流。

2.讲下布隆过滤器的原理吧  
答：布隆过滤器本质是一个位数组 + 多个哈希函数。插入元素时把多个哈希位置置 1，查询时只要有一个位置为 0 就一定不存在；如果都为 1，只能说明“可能存在”，会有误判但不会漏判。

3.滑动窗口限流，漏桶，令牌桶分别适合哪一种场景，每个说一个适用场景，并讲一下为什么适合  
答：  
滑动窗口适合接口级精细限流，比如“1 分钟内最多 100 次”，因为它统计精度高；  
漏桶适合稳定输出速率的场景，比如下游系统承压固定，希望请求匀速流出；  
令牌桶适合允许瞬时突发的场景，比如用户请求平时不高，但偶尔允许短时峰值，只要长期平均速率受控即可。  

# 算法题：
找一个字符串的最长回文子串  
答：经典做法有中心扩展和动态规划。面试里优先写中心扩展，代码更短、复杂度 O(n²)，从每个字符和字符间隙向两边扩展，实时更新最长回文区间即可。
