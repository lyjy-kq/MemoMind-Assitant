# 字节跳动Data后端二面

2026.4.28

实习拷打20分钟

- [ ] binlog怎么做数据迁移 结合实习问的  
答：标准方案是“全量 + 增量”。先做一次全量导出导入，把基础数据迁过去；然后从某个 binlog 位点开始订阅增量变更，持续回放 insert/update/delete，等延迟追平后切流。工程里要重点保证幂等、顺序和校验，比如按主键 upsert、记录位点、做数据对账。

- [ ] redis是单线程吗  
答：如果说命令执行模型，Redis 核心命令处理长期是单线程的；但从 Redis 6 开始，网络读写等 IO 部分可以多线程。面试时最好别简单回答“是”或“不是”，而要说清“命令执行单线程，部分 IO 多线程”。

- [ ] 为什么单线程还这么快  
答：因为它主要跑在内存里，命令路径短，数据结构针对性强，而且单线程避免了锁竞争和线程切换开销。另外它用 IO 多路复用高效处理大量连接，所以整体吞吐依然很强。

- [ ] kafak的熔灾与负载均衡  
答：这里一般可以按“容灾 + 均衡”两层讲。容灾依赖副本机制、ISR、leader 选举和跨 broker 分布；负载均衡体现在分区分布、消费者组再均衡、生产端按 key 或轮询把流量打散。再往深一点可以补一句：副本数、分区数、ack 策略和消费者并发度要一起设计，不是单点参数优化。

- [ ] kafak和rocketmq的区别  
答：Kafka 更偏日志流平台，吞吐高、生态成熟，适合大数据和流式处理；RocketMQ 在事务消息、延时消息、顺序消息和业务消息场景里更常见。二者本质都能做 MQ，但设计侧重点不同。

- [ ] 消息堆积如何处理  
答：先区分是生产太快、消费太慢，还是下游依赖异常。处理手段通常包括扩容消费者、增加分区、做消息降级或过滤、提升消费逻辑效率、把慢操作异步化；如果已经堆积严重，还要评估是否允许跳过低价值消息或离线补处理。

- [ ] k8s的常用组件介绍一下  
答：控制面常见有 API Server、etcd、Scheduler、Controller Manager；节点侧主要是 kubelet、kube-proxy 和容器运行时。回答时可以顺带说 Pod、Deployment、Service、Ingress 是最常用的工作负载和服务抽象。

- [ ] llm常见的算法了解哪些  
答：可以从预训练、微调、对齐三个阶段讲。预训练常见自回归 Transformer，微调有 SFT、LoRA、QLoRA，对齐常见 RLHF、DPO；检索增强、工具调用、Agent 则更多是系统工程层的增强。

- [ ] transformer了解吗  
答：Transformer 的核心是自注意力机制，用注意力替代 RNN 的顺序递归，便于并行训练和建模长距离依赖。标准结构包括多头注意力、前馈网络、残差连接、LayerNorm 和位置编码。

- [ ] agent知道多少  
答：Agent 不只是“模型会调用工具”，而是“模型在目标驱动下做规划、执行、观察和反思”的完整闭环。结合这个项目，可以说 `UnifiedAgent` 负责路由，复杂场景下 `runReAct` + `GraphRuntime` 会把任务拆成 DAG 节点，支持工具调用、子 Agent、重试、取消、快照和 replan，这才比较接近真正可落地的 Agent。

- [ ] harness工程 prompt工程 上下文工程的区别  
答：这题非常适合结合项目说。  
`prompt 工程` 更偏“怎么写指令”，比如 system prompt、few-shot、输出格式约束；  
`上下文工程` 更偏“给模型喂什么”，这个项目里就是 `promptctx` 按槽位装 profile、task memory、tool state、constraints、recall，并控制 token budget；  
`harness 工程` 则是模型外围执行框架，解决稳定性和闭环问题，比如这里的 `runOnce`、`GraphRuntime`、重试、超时、快照、取消、事件流、sandbox 安全边界。  
一句话概括：prompt 决定“怎么说”，context 决定“带什么料”，harness 决定“系统能不能稳定跑完”。

- [ ] 你的rag怎么做的   
答：这个项目的 RAG 不是单路向量检索，而是三路混合。文档 ingest 时先做 parent-child 分块；查询时先做 history-aware 改写，再走 Milvus 语义检索、Elasticsearch BM25 和 Neo4j 图检索，之后用 RRF 融合，必要时 rerank，再回填父块给 LLM 生成答案。这样能同时兼顾语义相似、关键词命中和关系扩展。

- [ ] 有没有遇到过幻觉问题 怎么解决的  
答：有，尤其是检索命中不稳、上下文过长或模型在证据不足时硬答的时候。我的处理思路是三层：第一层提升召回质量，比如 query rewrite、混合检索、rerank；第二层收紧生成约束，让回答尽量基于检索片段和工具 observation；第三层在工程上做“没证据就承认不确定”的兜底，而不是强迫模型每次都给完整答案。

- [ ] 你的rag怎么评估的  
答：我会拆成检索层和生成层。检索层看 Recall@K、MRR、nDCG、命中率、空召回率、延迟；生成层看答案正确性、faithfulness、来源覆盖率、人工抽样评分。结合这个项目，还可以做 ablation，比如对比只用向量检索和三路混合检索、对比开关 rewriter/reranker 后的效果差异。

- [ ] 算法k个一组反转链表  
答：标准做法是先检查剩余节点是否够 k 个，够的话反转这一段，再把前一段尾巴接上反转后的头，最后把当前段的尾巴接到下一段。实现时通常会引入 dummy 节点，便于处理头结点变化。
