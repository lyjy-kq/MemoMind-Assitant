# 商汤科技 后端研发

2026年1月

### 一面
25分钟项目 + 15分钟八股 + 20min算法

1.自我介绍  
答：可以直接按“技术栈 + 项目 + 负责部分”说。我会介绍自己主要做 Go 后端开发，最近做的是一个多模态智能体系统，负责统一 Agent 调度、RAG 检索、上下文拼装、工具执行和记忆系统这些核心链路。

2.介绍一下实习期间的难点 15min  
答：可以讲这个项目里“功能复杂但要稳定”的矛盾。比如难点不是单纯接 LLM，而是要把 `chat / tool / rag / react` 四种模式统一到 `UnifiedAgent.runOnce`，同时还要处理取消、超时、快照恢复、记忆写入和外部依赖降级；这类回答比只说“我写了个接口”更有说服力。

3.介绍一下channel  
答：channel 是 Go 里 goroutine 之间通信和同步的核心原语，强调“通过通信共享内存”。它支持无缓冲和有缓冲两种模式，常用于任务传递、结果汇总、限流、关闭通知。

4.介绍一下context包  
答：`context` 用来在调用链上传递取消信号、超时控制和请求级元数据。这个项目里它贯穿很深：HTTP 请求进来后会进入 `ProcessContext`，再传给工具执行、子 Agent、GraphRuntime，取消请求时可以一起停掉整条执行链。

5.如何防止goroutinue泄漏  
答：关键是让 goroutine 有退出条件，而且退出条件能被上层控制。实践上通常要配合 `context`、`select`、超时、关闭 channel 和避免阻塞发送；在这个项目里，长链路都挂了 `context.WithCancel` 和 step timeout，不会让工具执行无限挂住。

6.redis使用过哪些数据结构？  
答：最常见的是 string、hash、list、set、zset，另外 bitmap、geo、stream 也经常问。回答时最好顺带说场景，比如 zset 做排行榜，hash 存对象字段，stream 做消息流。

7.介绍一下innodb的索引b+树  
答：InnoDB 的索引底层是 B+ 树，叶子节点按顺序链起来，适合范围查询和排序。主键索引的叶子节点存整行数据，二级索引叶子节点存索引列和主键值，所以二级索引查整行常常要回表。

8.索引失效的场景  
答：常见有联合索引不满足最左前缀、对索引列做函数或表达式运算、隐式类型转换、`like '%xx'`、范围查询后后续列利用不足、数据分布极端导致优化器放弃索引。面试时别只背名词，最好补一句“最终结果就是走全表或低效扫描”。

分库分表的时候id怎么处理？  
答：常见方案有雪花算法、数据库号段、Leaf、UUID。真实工程里一般更偏向趋势递增且全局唯一的方案，比如雪花算法，既方便水平扩展，也兼顾索引局部性。

9.你的AI项目历史上下文有没有做截断处理  
答：有，而且是分层做的。这个项目里主对话历史由 STM 管理，RAG 改写只取最近 6 条消息，也就是最多 3 轮对话，代码在 `internal/application/chat/ctx_builder.go`；另外 `promptctx` 里每个槽位也有独立 budget，避免上下文无限膨胀。

10.你的propmt是怎么处理的  
答：不是简单拼一个大字符串，而是做了 schema-driven 的上下文装配。项目里 `promptctx` 会按 mode 把 profile、planner、task memory、tool state、constraints、recall 这些槽位并发取回，再按优先级和预算拼成最终 system prompt，这样 prompt 工程就从“手写模板”升级成“结构化上下文工程”。

11.平时使用哪些AI编程工具？？  
答：可以说我会用 Claude Code、Cursor、Copilot 这类工具，但重点不是“列工具名”，而是“怎么用”。比如我会让它们帮我做代码检索、测试样例生成、重构建议和文档整理，但关键逻辑、边界条件和技术决策一定自己复核。

算法 lc LCR 087复原IP地址（思路对的边界处理有问题，估计挂了）  
答：标准做法是回溯。每次切 1 到 3 位，判断是否前导零、是否大于 255，累计切出 4 段且正好用完整个字符串时加入结果，时间复杂度可以近似看成常数级分支搜索。

### 二面：
你认为go和python哪个更适合做大模型？  
答：如果是训练和算法研发，Python 生态更成熟；如果是做大模型服务化、Agent 编排、网关、检索、工具执行这些工程落地，Go 更适合高并发和部署。这个项目就是典型例子：模型调用只是其中一环，真正复杂的是 RAG、记忆、路由、工具和稳定性治理，这些用 Go 做后端服务很合适。

实习15分钟  
答：建议回答时按“背景-目标-方案-难点-结果-复盘”讲，不要流水账。最好抓住一个有取舍、有稳定性挑战的点来展开，比如为什么要把上下文工程从简单拼接改成 schema-driven。

讲解一下HTTP和TCP的区别  
答：TCP 是传输层协议，负责可靠传输；HTTP 是应用层协议，定义请求响应语义。HTTP 基于 TCP 之上，前者解决“传什么、怎么表达”，后者解决“怎么可靠送到”。

如何理解HTTP的无状态  
答：无状态是指协议层不自动记住前一次请求的业务上下文。服务端如果要维持登录态、会话态，需要借助 cookie、session、token 或者应用层存储。

为什么我们常说HTTP是短连接 TCP是长连接  
答：这是早期 HTTP/1.0 时代的常见说法，不够严格。TCP 本身是面向连接的，只要不四次挥手就算连接还在；HTTP 是否复用连接要看版本和头部设置，HTTP/1.1 默认 keep-alive，HTTP/2 更强调多路复用。

浏览器DNS啥的  
答：输入 URL 后通常会经历浏览器缓存检查、DNS 解析、TCP 三次握手、TLS 握手、HTTP 请求、服务端处理、响应渲染。面试里可以按这条主链讲，不要东一榔头西一棒子。

RESTFULL  
答：RESTful 强调资源化表达和统一接口风格，比如资源用名词、动作落在 HTTP 方法上、无状态、幂等语义明确。比如 `GET /documents/{id}` 查资源，`POST /documents` 创建资源，这个项目的文档接口就是这种风格。

GIN框架的middleware  
答：本质是责任链，前置处理和后置处理围绕 handler 执行，常用于日志、鉴权、恢复、限流。虽然这个项目用的是 `chi`，但思路一样，像 `RequestID`、`PanicRecover`、`AccessLog`、`CORS` 都是中间件。

gorm框架  
答：GORM 是 Go 里常用 ORM，优点是开发效率高、模型映射方便，缺点是复杂 SQL 和性能优化时容易失控。这个项目反而没有走 GORM，而是 repo + 持久化实现分层，原因是这里对 PostgreSQL、Milvus、ES、Neo4j 这些异构存储的控制要求更高。
