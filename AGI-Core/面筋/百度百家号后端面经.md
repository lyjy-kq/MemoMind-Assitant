# 百度百家号后端面经

时间：3月中旬

### 一面：
1.常见的设计模式  
答：常见的有单例、工厂、策略、责任链、观察者、模板方法。结合这个项目，`toolRegistry` + 不同工具实现属于策略模式，`sandbox` 按 backend 选择 Docker/Local/Mock 属于工厂模式，HTTP 中间件链和 ReAct 执行链都可以看成责任链模式。

2.MySQL增删改查语句  
答：增是 `insert`，删是 `delete`，改是 `update`，查是 `select`。面试里最好顺带说清楚条件过滤、排序、分页、聚合和 join，例如 `select ... from ... where ... order by ... limit ...`，这样更像真实后端开发。

3.MySQL事务  
答：事务就是把一组操作放在一个原子执行单元里，要么全部成功，要么全部回滚。它的核心特性是 ACID，InnoDB 主要通过 undo log、redo log、锁和 MVCC 来保证事务能力。

4.隔离级别  
答：从低到高是读未提交、读已提交、可重复读、串行化。MySQL InnoDB 默认是可重复读；读已提交主要解决脏读，可重复读进一步解决不可重复读，串行化最强但并发最低。

5.索引种类  
答：按功能分有主键索引、唯一索引、普通索引、联合索引；按数据组织分有聚簇索引和二级索引；按数据结构常说 B+ 树索引、哈希索引。真实场景里最常用的是 InnoDB 的 B+ 树主键索引和联合二级索引。

6.b➕树特性  
答：B+ 树非叶子节点只存索引键，不存完整数据，叶子节点存完整键值并且通过链表串起来，所以范围查询、排序、分页都很友好。它的分支因子大、树高低、磁盘 IO 次数少，这也是 MySQL 选它做默认索引结构的原因。

7.Redis数据结构  
答：常见有 string、list、hash、set、zset、bitmap、hyperloglog、geo、stream。面试时最好顺带补一句使用场景，比如 string 做缓存、hash 存对象、zset 做排行榜、stream 做轻量消息流。

8.Redis为什么快  
答：一是基于内存，二是单线程事件循环避免了大量锁竞争，三是 IO 多路复用提升了网络处理效率，四是底层数据结构做了很多针对性优化。再往项目里落，可以说这种“少锁 + 内存计算 + 明确数据结构”的思路和我们项目里 `toolRegistry`、`memoryStack` 做并发隔离的思路是一致的。

9.创建线程的方式（差点忘了，面试管提醒了一手）  
答：如果按 Java 说，常见是继承 `Thread`、实现 `Runnable`、实现 `Callable + Future`、交给线程池执行。要是面试官问 Go，就可以直接回答 Go 里一般不用“手动创建线程”，而是启动 goroutine，由 GMP 调度模型复用底层线程。

10.线程池好处  
答：核心好处是复用线程、降低频繁创建销毁开销、控制并发上限、统一排队和拒绝策略、便于监控。工程上它本质上是在吞吐、延迟、资源占用之间做平衡。

11.线程池原理，7大参数  
答：原理是“核心线程 + 阻塞队列 + 最大线程数 + 拒绝策略”。`ThreadPoolExecutor` 七个关键参数是 `corePoolSize`、`maximumPoolSize`、`keepAliveTime`、`timeUnit`、`workQueue`、`threadFactory`、`RejectedExecutionHandler`；执行流程通常是先用核心线程，再进队列，队列满了再扩容到最大线程，最后触发拒绝策略。

12.Linux命令排查  
答：常用思路是先看进程和资源，再看网络和日志。比如 `top/htop` 看 CPU 和内存，`ps -ef | grep` 看进程，`ss -lntp` 或 `netstat` 看端口，`tail -f` 看日志，`df -h` 看磁盘，`free -h` 看内存，`lsof -i :端口` 看端口占用；如果排查本项目，还会重点看 `/healthz`、`/readyz` 以及基础设施连接状态。

### 二面：
问了实习和项目，问题就在于技术选型以及一些ai的问题  
答：这类题建议直接围绕项目讲三层。

第一层讲技术选型：项目用 Go 做后端，`chi` 做 HTTP 路由，原因是启动快、并发模型轻、部署简单；数据层用 PostgreSQL 做持久化，Milvus/ES/Neo4j 做 RAG 三路检索，原因是分别覆盖语义、关键词、图关系三类召回能力；同时所有外部依赖都允许降级，保证系统可用性。  

第二层讲 AI 主链路：请求进入后由 `UnifiedAgent` 做路由，纯对话走 chat，知识问答走 RAG，复杂任务走 ReAct；RAG 侧会先做 query rewrite，再走 Milvus + ES + Neo4j 混合检索，RRF 融合后回填父块给 LLM 生成答案。  

第三层讲工程稳定性：项目里把 prompt 工程、上下文工程、执行工程拆开了。`promptctx` 负责组装 profile/task/tool/constraints/recall 等上下文槽位，`runtime_graph` 负责 DAG 执行、超时、重试、快照和取消，`sandbox` 负责命令执行安全边界，这样项目不是“能跑就行”，而是“能持续稳定跑”。
