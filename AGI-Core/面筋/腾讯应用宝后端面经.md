# 腾讯应用宝后端面经

时间：2026年3月中到3月末

前情提要：大部分关于实习的面经的参考意义不大，所以这次只记录1-3面所涉及的八股文。

### 一面：
1.redis数据结构  
答：常见有 string、list、hash、set、zset、bitmap、hyperloglog、geo、stream。回答时最好补使用场景，比如 zset 做排行榜，hash 存对象，stream 做消息流。

2.redis持久化机制  
答：Redis 有 RDB 和 AOF 两套主流持久化机制。RDB 是按时间点生成快照，恢复快但可能丢最近一段数据；AOF 是追加写命令日志，数据更完整，但体积更大、恢复更慢。

3.mysql索引底层  
答：InnoDB 索引底层主要是 B+ 树。它让非叶子节点只存键，叶子节点有序且链表相连，所以磁盘 IO 少、范围查询和排序效率高。

4.聚簇索引与非聚簇索引  
答：聚簇索引的叶子节点直接存整行数据，InnoDB 主键索引就是聚簇索引；非聚簇索引或二级索引叶子节点存索引列和主键值，需要再回主键索引取整行。查询是否回表，很多时候就决定了性能差异。

5.索引优化  
答：核心是让查询尽量走合适的索引并减少回表。常见方法有建立高区分度索引、按查询条件顺序设计联合索引、覆盖索引、避免 select *、避免在索引列上做函数和隐式转换。

6.索引失效  
答：典型场景有最左前缀不满足、函数或表达式作用在索引列上、隐式类型转换、`like '%xx'`、优化器评估后认为全表扫描更便宜。面试里最好讲“原因 + 结果”，不要只背触发词。

7.mysql执行一条sql  
答：大致流程是连接器、解析器、优化器、执行器。到 InnoDB 层后还会涉及索引选择、Buffer Pool、锁、undo/redo、binlog 等，更新语句会先写内存页和日志，再按时机刷盘。

8.那么多索引mysql怎么选  
答：优化器会基于统计信息估算不同执行计划的成本，比如扫描行数、回表成本、是否排序、是否临时表等，然后选成本更低的计划。统计信息不准时就可能出现“明明有索引却没选”的情况。

9.tcp与udp区别  
答：TCP 面向连接、可靠传输、有序、带流量控制和拥塞控制；UDP 无连接、尽力而为、开销小、实时性好。前者适合大部分业务请求，后者适合实时音视频、DNS 这类对时延更敏感的场景。

10.tcp为什么可靠  
答：因为它有序列号、确认应答、超时重传、滑动窗口、流量控制、拥塞控制，还能校验数据完整性。简单说就是既保证“送达”，也尽量保证“按顺序、不过量地送达”。

11.消息队列作用  
答：主要是解耦、削峰填谷、异步化、广播和最终一致性。像这个项目里的事件总线就可以把聊天事件、RAG ingest 事件和业务主链路解耦开。

12.kafka怎么保证消息有序性  
答：Kafka 只能保证分区内有序，不能天然保证全局有序。要保证某类消息顺序，通常做法是同一个业务 key 落同一分区，生产端按 key 路由，消费端单线程或有序处理该分区。

13.mcp是什么？  
答：MCP 是 Model Context Protocol，本质上是模型和外部工具、资源之间的标准化协议。它解决的是“模型怎么稳定发现工具、调用工具、读取资源”的问题，和这个项目里 `toolRegistry` 管理工具的思路类似，但 MCP 的目标是跨平台标准化。

14.skills是什么？  
答：skills 可以理解成可复用的高层工作流或操作规范，不只是一个函数调用，而是一整套“在什么场景下怎么做”的封装。比如代码审查、文档整理、项目导读，都可以沉淀成 skill。

15.jvm内存分配与回收过程  
答：对象通常先分配在 Eden，经过 Minor GC 后进入 Survivor，达到年龄阈值再晋升到老年代；大对象也可能直接进老年代。GC 会根据分代特点采用复制、标记清除、标记整理等策略。

16.fullgc触发机制  
答：常见触发包括老年代空间不足、`System.gc()`、元空间不足、CMS/G1 某些回收失败后的降级等。Full GC 停顿重、代价大，线上通常要尽量避免频繁触发。

17.tcp的拥塞控制流程  
答：核心阶段是慢启动、拥塞避免、快速重传、快速恢复。开始时拥塞窗口指数增长，达到阈值后线性增长；如果检测到丢包，就缩小窗口并进入相应恢复阶段。

18.分布式事务解决方案，说了2pc，3pc，tcc。  
答：2PC 强一致但阻塞明显，协调者单点问题重；3PC 在 2PC 上尝试降低阻塞，但实现复杂、落地少；TCC 是业务层补偿方案，Try/Confirm/Cancel 侵入业务但更灵活。互联网业务里常常还会用本地消息表、最终一致性来折中。

### 二面：
1.redis与数据库的缓存一致性  
答：常见方案是 Cache Aside，先更新数据库再删除缓存，而不是先更新缓存。严格一致很难，一般通过延迟双删、消息通知、版本号、幂等重试把不一致窗口压到可接受范围。

2.redis的zset数据结构  
答：ZSet 底层一般由跳表 + 字典组成。字典负责按 member 快速定位，跳表负责按 score 排序和范围查询，所以它既能查单点，也能做排行榜。

3.redis的两个持久化机制  
答：RDB 和 AOF。线上往往组合使用，RDB 负责快速恢复基线，AOF 负责减少数据丢失窗口。

4.分布式cap理论  
答：CAP 指一致性、可用性、分区容错性。网络分区发生时，系统不可能同时完美满足 C 和 A，只能在 CP 和 AP 之间取舍。

5.什么时候ap，什么时候cp  
答：涉及金融扣款、库存扣减、强一致配置这类核心正确性问题更偏 CP；涉及推荐、搜索、评论计数、缓存副本这类允许短暂不一致的场景更偏 AP。选型本质上是业务容错度决定的。

6.进程线程协程  
答：进程是资源分配单位，线程是 CPU 调度单位，协程是用户态轻量执行单元。Go 的 goroutine 本质上就是用户态协程，由 GMP 模型调度到内核线程上运行。

### 三面：
1.学校都有什么课程  
答：建议挑和岗位相关的讲，比如数据结构、操作系统、计算机网络、数据库、编译原理、软件工程，再补一句自己如何把这些基础落到项目里。

2.spring的ioc原理以及优点  
答：IoC 是控制反转，核心是对象依赖不由业务代码手动 new，而交给容器管理。好处是解耦、方便扩展、易测试，配合 DI 可以把依赖关系显式化。

3.除了解耦还知道什么？  
答：还包括提升可测试性、统一生命周期管理、便于 AOP、降低配置和装配复杂度。面试里只说“解耦”会显得太薄。

4.springboot与spring区别，二者的源码看过没？Tomcat了解嘛？有没有具体看过  
答：Spring Boot 是在 Spring 基础上做自动配置、约定优于配置和嵌入式容器整合，降低启动和装配成本。Tomcat 可以从 connector、container、pipeline、request 生命周期这些角度概括；如果没深看源码，就诚实说看过启动流程和部分组件即可。

5.spring的bean，面试官一直在重复一个思想问我懂不懂，完全没听过  
答：大概率是在追问生命周期、作用域、依赖注入或者三级缓存解决循环依赖。回答 bean 时可以主动讲实例化、属性填充、初始化、销毁，以及 singleton/prototype 等作用域。

6.mybatis是干什么的？ibatis用过没？平常怎么写SQL？完全不写嘛？  
答：MyBatis 是半自动 ORM，更强调 SQL 可控性；iBatis 是它的前身。后端面试里最好明确表态：复杂查询、索引优化、批处理场景一定要能自己写 SQL，而不是完全依赖 ORM。

7.设计一个分布式双十一秒杀系统  
答：可以从流量入口、限流削峰、库存预扣、异步下单、幂等、MQ、缓存、热点隔离、监控告警来讲。核心目标是“先保系统不崩，再保库存不超卖，再优化用户体验”。

8.怎么做限流  
答：可以按网关、接口、用户、IP、业务资源多维限流。算法上常见令牌桶、漏桶、固定窗口、滑动窗口；工程上要结合熔断、降级、排队一起看。

9.缓存与数据库一致性，你做异步要用户等你嘛？  
答：通常不会让用户同步等所有异步链路完成，而是把主交易路径和次要同步路径拆开。主链保证写 DB 成功，缓存更新、搜索索引更新、通知这类走异步补偿，但关键状态要有幂等和兜底重试。

10.负载均衡怎么做  
答：可以在 DNS、SLB、网关、服务发现、消费者侧多层做。常见算法有轮询、加权轮询、一致性哈希、最少连接数，关键是结合健康检查和熔断摘除。

11.多数据中心还是单数据中心，如果出现没卖完怎么做  
答：如果业务更看重一致性，先做单数据中心更容易控风险；如果是超大规模高可用场景，再做多活，但代价是库存一致性复杂度显著上升。没卖完要区分是库存预热、缓存扣减、订单确认哪个环节出了偏差，再通过对账和补偿修正。

12.平常读书吗？  
答：可以答“会读技术书，也会读一些非技术类书”，然后举 1 到 2 本真正读过的，别空泛。

13.上过哲学课嘛？  
答：可以从“哲学训练抽象能力、问题拆解和反思能力”切入，不用装懂太多流派。

14.兴趣爱好有没有  
答：尽量讲能体现持续投入和节奏感的爱好，比如跑步、羽毛球、阅读、写技术笔记，而不是只说“打游戏”。

15.对ai的看法  
答：可以说 AI 已经从“辅助问答”进入“参与软件生产流程”的阶段，但它更像放大器，不会替代工程判断。结合这个项目，还可以补一句我更关注 AI 系统化落地，而不是只做 prompt demo。

16.来深圳有问题嘛？  
答：直接正面回答是否能接受，并简要说明安排即可。

17.为什么不考研  
答：重点不是解释“没考”，而是解释“为什么选择直接做工程实践”。可以说自己更偏向在真实系统里提升架构、性能和业务抽象能力。

18.上大学带给了你什么？你提升在哪里，有没有具体的例子？  
答：可以从学习能力、工程协作、问题拆解、抗压和表达能力来讲，再举一个项目例子，比如把这个智能体项目从“能调用模型”推进到“有检索、有记忆、有执行稳定性”的过程。
